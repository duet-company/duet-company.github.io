### Problem
ClickHouse container fails to initialize in GitHub Actions CI because it detects existing database files and skips initialization.

### Solution - New Approach
Instead of relying solely on tmpfs, this PR uses an **entrypoint override** to explicitly clear data directories before ClickHouse starts:

```yaml
options: >-
  --entrypoint bash
  -c
  rm -rf /var/lib/clickhouse/* /var/log/clickhouse-server/* && exec /entrypoint.sh docker-entrypoint.sh
  --health-cmd "curl -f http://localhost:8123/ping || exit 1"
  --health-interval 30s
  --health-timeout 30s
  --health-retries 20
  --health-start-period 180s
  --tmpfs /var/lib/clickhouse:rw,noexec,nosuid,nodev
  --tmpfs /var/log/clickhouse-server:rw,noexec,nosuid,nodev
```

### Key Changes
1. **Entry Point Override:** Explicitly clear directories before starting
2. **Version:** Downgrade to ClickHouse 23.3 (stable)
3. **Tmpfs:** Still used for ephemeral storage (defense in depth)
4. **Timeouts:** Extended health checks (180s start period)

### Why This Should Work
- The `rm -rf` command runs **before** ClickHouse entrypoint checks for existing data
- This forces ClickHouse to reinitialize even if it thinks data exists
- Tmpfs provides additional protection against data persistence
- ClickHouse 23.3 is proven stable in CI environments
