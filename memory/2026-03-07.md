## Afternoon Investigation (2:03 PM Europe/Berlin)

### Deep Dive: CI Failure Root Cause

**Issue Analysis:**
PR #14 is failing because ClickHouse initialization is detecting an existing database and skipping initialization:

```
/entrypoint.sh: create new user 'default' instead 'default'
ClickHouse Database directory appears to contain a database; Skipping initialization
Failed to initialize container clickhouse/clickhouse-server:23.3
```

**Problem:**
- tmpfs mounts are configured: `--tmpfs /var/lib/clickhouse:rw,noexec,nosuid,nodev`
- tmpfs mounts are configured: `--tmpfs /var/log/clickhouse-server:rw,noexec,nosuid,nodev`
- However, ClickHouse is STILL detecting a database directory
- This suggests tmpfs mount is not working correctly in GitHub Actions

**Secondary Issue:**
PostgreSQL health checks are failing with `FATAL: role "root" does not exist`
- This is because workflow doesn't specify `POSTGRES_USER`
- `pg_isready` health check tries to connect as "root" (system user)
- However, this is secondary - the real blocker is ClickHouse

**Key Finding:**
The tmpfs approach (PR #14) should work in theory, but GitHub Actions Docker service containers may not support tmpfs mounts the same way as regular `docker run` commands.

**Previous Attempts All Failed:**
- PR #10: ClickHouse 23.3 + extended timeouts → FAILING (database corruption)
- PR #11: ClickHouse 23.3 + tmpfs + entrypoint override → FAILING
- PR #12: ClickHouse 23.3 + entrypoint wrapper → FAILING (syntax error)
- PR #13: Entry point syntax fix → FAILING (Docker parses `-c` as cpu-shares)
- PR #14: ClickHouse 23.3 + tmpfs (no entrypoint) → FAILING (database persists)

**Root Cause:**
GitHub Actions service containers may not support all Docker options, particularly tmpfs mounts in the same way as interactive containers. The tmpfs is being ignored or not properly mounted, causing data persistence between runs.

**Potential Solutions to Investigate:**

1. **Use volume cleanup in workflow:**
   - Add step before tests to manually clean ClickHouse data
   - Use `docker exec` to remove database files

2. **Try different health check:**
   - Current: `curl -f http://localhost:8123/ping`
   - Alternative: Use ClickHouse client for health check
   - Or: Disable health check entirely and use retry in tests

3. **Use init container:**
   - Separate init container that prepares ClickHouse
   - Then mount that as volume for test container

4. **Change CI strategy:**
   - Move away from GitHub Actions service containers
   - Use self-hosted runners or different CI platform
   - Or: Use docker-compose within workflow step

5. **Simplify: Remove ClickHouse from CI temporarily:**
   - Mock ClickHouse for unit tests
   - Run integration tests separately (manual or different pipeline)
   - Allows unblocking PRs #6 and #7

**Recommended Next Steps:**

**Immediate (Today):**

   - Modify test suite to use mock ClickHouse for CI
   - Run real ClickHouse tests locally or in separate pipeline
   - Merge PRs #6 and #7 (unblocks 3,725 lines of code)

## Evening Update (10:03 PM Europe/Berlin)

### Circular Import Fix Applied

**Discovery:** PR #15 CI failing with circular import error
- Error: `ImportError: cannot import name 'get_password_hash' from partially initialized module 'app.core.security'`
- Root cause: Circular import chain between auth and core modules

**Circular Import Chain:**
1. `app/auth/__init__.py` → `app.auth.routes`
2. `app/auth/routes.py` → `app.auth.service`
3. `app/auth/service.py` → `app.core.security` (imports get_password_hash, verify_password, etc.)
4. `app/core/security.py` → `app.auth.schemas` (imports TokenData) ← **Creates cycle**

**Solution Applied:**
- Moved `TokenData` schema from `app/auth/schemas.py` to `app/core/security.py`
