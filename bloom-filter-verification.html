Optimization Details

Before
Single Bit Hash

1 bit per hash → 1 memory access per check

After
Dual Bit Hash

2 bits per hash, packed in same uint32 → still 1 memory access

Bit Allocation (32-bit hash)

  • 16 bits → array index
  • 5 bits → first bit position (0–31)
  • 5 bits → second bit position (0–31)
  • 6 bits → unused

Mathematical Verification

Standard bloom filter false positive rate formula:

(1 - e^(-kn/m))^k

Parameters: n = 256,000 elements, m = 2,000,000 bits

  • k = 1: 11.68% FPR ✓
  • k = 2: 5.69% FPR ✓

The claimed 2.1× reduction is mathematically correct.

Trade-offs & Limitations

Documented Trade-offs

  • Bit independence loss: two bits derived from same hash → slight collision increase
  • Marketing framing: "2x more accurate" vs "~2x lower false positive rate"
  • Benchmark environment: performance varies by hardware and workload

The authors correctly identify that avoiding an extra memory access in performance-critical paths justifies the theoretical trade-off.

Why It Works

  • Cache locality: Single uint32 access preserves L1/L2 cache efficiency
  • Lock-free atomicity: One atomic OR sets both bits, no contention
  • CPU cost: Extra bit shift (few cycles) vs memory access (hundreds of cycles)

Conclusion

This is excellent applied engineering, not theoretical innovation. The optimization provides measurable value in production systems handling large-scale data scanning, storage engine efficiency, and hash join performance.
